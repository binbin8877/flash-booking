# AI 활용 기록

*결과물* 보다 *판단 과정* 의 기록.
모든 AI 산출물은 *이해 → 검증 → 본인 판단* 사이클로 거른 후 채택 / 거절.

## 사용한 도구

| 도구 | 용도 |
|---|---|
| Claude (plan mode) | 초기 설계 의뢰 — 요구사항 명세 기반 |
| Claude (대화) | 코드 생성 / 디버깅 / 설계 토론 |

---

## Phase 1. 설계 토론

plan mode 의 초기 설계안에 대해 의문 / 본인 안 제시 / 토론 사이클로 다듬음.

| 본인 의문 / 역제안 | 도출 결과 |
|---|---|
| Java 17 → 상위 버전 검토 의문 | Java 21 채택 |
| 본 프로젝트에서 Virtual Thread 적합성 의문 | 부적합 결론 → RateLimiter + Redis Lua 정당화 |
| sweeper 1분 주기의 정합성 의문 | event-driven primary + polling backstop 도출 |
| Redis 다운 시나리오 부하 테스트 부재 지적 | Chaos load test 신설 |

---

## Phase 2. 코드 이해 + 정합성 검증

생성된 코드를 한 줄씩 의미 검증 후 채택. 모순 / 누락은 즉시 정정 요청.

### 2-1. 주요 기술 결정의 근거 확인

| 본인 질문 | 영역 |
|---|---|
| sealed / pattern matching 의 본 프로젝트 적용처 | Java 21 신기능 |
| SKIP LOCKED / GET_LOCK / advisory lock 의 선택 근거 | MySQL 락 메커니즘 |
| reserve / release / reconcile 3종 Lua 분리 근거 | Redis 원자 연산 |

### 2-2. 정합성 / 상태 이상 발견 → 수정

| 본인 발견 | 결과 |
|---|---|
| 아키텍처 가정이 실제 단일 MySQL / Redis 구성과 불일치 | 토폴로지 정정 |
| hold TTL ↔ sweeper 주기 mismatch (4분 보상 윈도우 공백) | TTL ↔ 주기 정렬 |

---

## Phase 3. 추천 → 본인 판단 → 수용 / 거절

### 3-1. 옵션 비교 + 본인 판단으로 채택

| 상황 | 본인 판단 |
|---|---|
| sweeper 3 안 (A/B/C) 비교 | 각 안의 트레이드오프 확인 후 C 안 채택 |
| Java 25 도입 검토 | 검증 후 거절 — LTS 유지 |

### 3-2. 시니어 시뮬레이션 — 트레이드오프 지점

기술 / 사용자 / 운영 사이의 트레이드오프 지점에선 *"시니어 백엔드 개발자"* 관점을 시뮬레이션.

| 트레이드오프 축 | 적용 영역 |
|---|---|
| polling vs event-driven | sweeper 구조 |
| 결제 여유시간 vs 재고 드리프트 윈도우 | hold TTL 값 |
| 운영 부담 vs 정합성 즉시성 | sweeper 주기 |
| 안정성 (LTS) vs 신기능 | Java 버전 |

---

## Phase 4. 검증

### 4-1. 요구사항이 진실원천 — AI 가정의 역추적

| 본인 행위 | 결과 |
|---|---|
| hold TTL 5분의 *사용자 체감* 근거 확인 | 명세에 결제 여유시간 요구 없음 → *sweeper backstop 정렬* 목적으로 의미 재정의 |
| 부하 가정 200 TPS 의 요구사항 근거 역추적 | 50 TPS 로 정정 |

### 4-2. 본인 직접 실행

| 본인 행위 | 결과 |
|---|---|
| `curl /actuator/health` | 헬스체크 확인 |
| `k6 run` (정상 / Chaos) | `created_total: 0` 비정상 결과 → 재고 리셋 누락 발견 |
| `./gradlew test` | 통합 테스트 8종 전수 통과 확인 |

## AI 가 틀린 / 헷갈리게 답한 사례 (본인이 잡음)

- **Outbox ↔ 2-phase 분리 혼동** — 둘은 다른 개념임을 정정
- **hold TTL ↔ sweeper 주기 mismatch** — 주기 5분 변경 후에도 TTL 60초 추천 → 4분 보상 윈도우 공백 짚어 300초로 정정
- **아키텍처 가정과 실제 구성 불일치** — App 2 노드 구성을 단일 노드로 가정 → 정정
- **들러리 옵션** — 옵션 비교 시 실질적 비교가 아닌 항목을 채워넣는 경향 → 진짜 비교만 남김
